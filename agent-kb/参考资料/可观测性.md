# 🔭 可观测性设计 (详细参考)

在复杂的 Agent 系统中，排查诸如“为什么大模型在第 15 轮对话时突然卡住了 30 秒”或者“工具调用的参数到底是哪里传错的”这样的问题，如果没有优秀的可观测性（Observability）支持，犹如蒙眼走钢丝。

`packages/agent/docs/observability.md` 定义了 Pi 系统如何实现一套**与环境解耦、与厂商无关**的强大监控追踪体系。

## 🎯 核心设计目标：不依赖 OTel 或 Sentry

Pi 的核心库必须能够在 Node.js, 浏览器, Bun 以及 Edge Workers 等各种极其苛刻的 JS 运行时中执行。
因此，Pi 绝对不能直接引入 `OpenTelemetry (OTel)` 或 `Sentry` 的 SDK。一旦引入，整个包将变得臃肿且与特定平台绑定。

**解决方案：Pi 只定义抽象的事件契约，把解释权交给宿主的适配器。**

## 🗺️ 心理模型：追踪 (Traces) 与 跨度 (Spans)

系统采用业界标准的追踪模型：
- **Trace (追踪)**: 贯穿整个请求生命周期的一棵巨大的因果关系树。例如：用户输入一次 Prompt 并最终得到完整回复的全过程。
- **Span (跨度)**: 树上的一个节点，代表一次有明确耗时的操作。它通过 IDs（而非对象引用）来描述父子关系。

```typescript
// 一个 Span 日志的抽象结构
interface SpanRecord {
  traceId: string;       // "t1"
  spanId: string;        // "s2"
  parentSpanId?: string; // "s1" (代表它是 s1 的子任务)
  name: string;          // "pi.ai.provider.request"
  startTime: number;
  endTime?: number;
  status: "ok" | "error";
}
```

## 🧵 并发难题：异步上下文传递 (Async Context)

在 Node.js 中，大量的 I/O 操作是并发交织在同一个线程里的。传统的做法是到处把 `context` 对象作为函数的参数传来传去（称为 Context Drilling），这会污染所有的业务 API。

为了解决这个问题，Pi 引入了针对不同运行时的底层环境适配：
- **Node.js 环境**: 利用内置的 `AsyncLocalStorage` (类似于 Java 的 ThreadLocal)。当一个 Trace 开始时，将 ID 放入 LocalStorage。在这个异步链条里深处被调用的任何函数（不管嵌套了多少层 Promise），都可以直接从空气中（LocalStorage）抓取到当前归属的 Trace ID，而不需要通过函数参数传递。
- **Browser/Worker 环境**: 采用局部降级方案或使用新兴的 JS 特性进行跨异步任务的上下文传播。

## 📍 核心仪器化点 (Instrumentation Points)

系统并不监控每一行代码，而是在极其关键的“血管动脉”上安装了探针。
当这些操作发生时，系统会标准化地吐出 `start`, `end`, 或 `error` 事件。

### 1. Harness 结构层事件
这类事件用于追踪系统的高层编排状态：
- `pi.agent.prompt`
- `pi.agent.skill`
- `pi.agent.compaction`
- `pi.agent.branch_navigation`
- `pi.agent.session.append_entry` (监控数据库 IO 性能极其有用)

### 2. AI 网络请求层事件
这类事件是定位大模型性能瓶颈的重中之重：
- `pi.ai.provider.request`

当它发出 `end` 事件时，会附带极具价值的 Payload：
- 状态码 (Status Code)
- 中止原因 (Stop Reason: 比如是因为超过了 MaxTokens，还是被用户手动 abort 了)
- 重试次数 (Retry Count)
- Token 消耗清单 (Input/Output/CacheRead Tokens)

## 🛡️ 数据安全与脱敏策略 (Redaction)

日志往往是泄露用户隐私的重灾区。Pi 在核心层就制定了极其严格的“默认安全”数据采集规范。

**✅ 默认允许被记录的（安全元数据）**：
- 厂商名字 (Provider)、模型 ID (如 gpt-4o)
- Session ID、操作耗时、状态码
- Token 计数和消费金额

**❌ 默认被屏蔽、绝对不会出现在追踪事件中的（敏感数据）**：
- 用户的提问文本 (Prompts)
- 大模型的生成文本 (Completions)
- 调用工具时传入的具体参数 (Tool Args)
- 文件系统读取的内容 (File Contents)
- API 密钥 (API Keys)

如果私有部署的客户确实需要留存 Prompt 用于审计，他们必须在宿主层的适配器中**显式开启 (Opt-in)** 并配置专用的脱敏钩子，核心库绝不越俎代庖。

## 📦 生态故事 (The Package Story)

基于上述设计，Pi 的可观测性演化成了极度解耦的组件结构：

1. **`packages/observability`**: 最纯粹的核心定义。定义了接口和 `traceOperation()` 辅助函数。
2. **`packages/agent` & `packages/ai`**: 核心业务包。它们依赖包 1，并在关键节点调用 `traceOperation` 埋点。
3. **`packages/observability-node`** (可选): 为 Node.js 提供的专属适配器。使用 `diagnostics_channel` 充当事件总线。
4. **外部集成 (如 `pi-otel-bridge`)**: 开发者可以自己写一个小脚本，监听上述的事件总线，把抓取到的事件转化为标准的 OpenTelemetry Span，发给 Datadog 或 Jaeger。核心系统对 OTel 一无所知。

> [!abstract] 最终结论
> Pi 定义了稳定、安全的**事件契约**。而各种适配器决定了这些事件**最终去向何方**。这种设计保证了核心库的轻盈与长久生命力。
