# 📖 AgentHarness 生命周期 (详细参考)

这份文档是对 [[核心概念/生命周期]] 的深度补充，主要面向需要在 Harness 级别编写复杂 Hook 插件、或者需要实现底层 Session 驱动的开发者。

## 🎯 终极生命周期目标 (The Ultimate Goal)

系统设计者的愿景非常明确：
**任何 Harness 的监听器和钩子，都应该能够安全地闭包引用 `AgentHarness` 实例，并在任何允许的事件回调中，直接调用 Harness 的公共 API。**

这就要求系统必须做到绝对的防弹（Bulletproof）。因为你永远不知道一个第三方的 Hook 会在 `tool_call` 事件里做什么。如果他在 `tool_call` 里调用了 `harness.setModel()`，系统不能死锁，不能破坏正在进行中的回合快照 (In-flight turn snapshots)，也不能把这条更改历史写错位置。

### 核心规则 (The Rules of Engagement)

1. **结构性操作的排斥性**: `prompt()`, `compact()`, `navigateTree()` 等涉及修改树状结构的 API，在 `busy` 状态下必须毫不留情地被拒绝（抛出异常）。
2. **队列操作的宽容性**: `steer()`, `followUp()`, `nextTurn()` 是永远被接受的。它们只是默默把数据推入队列，等待下一个安全的时间点（如回合开始或结束）被消费。
3. **运行时配置的未来性**: 调用配置 Setters（如 `setTools`, `setModel`）会立刻修改内存，但**绝不**突变当前正在发往 LLM 提供商的请求快照。它们只影响未来的回合。
4. **会话写入的确定性**: 在忙碌状态下产生的所有 Session 写入请求，必须被“挂起（Pending）”，并在确定的“保存点”按照先进先出的顺序一次性冲刷 (Flush)。
5. **死锁防范**: 事件监听器目前没有门面 (Facade) 保护。如果一个监听器闭包了 `harness` 并试图在回合还在进行时 `await harness.waitForIdle()`，就会产生死锁（它在等待自己所在的事件循环结束）。未来计划引入安全的 `runWhenIdle()` API。

## 🚨 错误处理策略分层 (Error Handling Split)

系统根据所处层级的不同，采用了截然不同的错误处理流派：

### 底层能力层：Rust 风格的 `Result`
对于文件系统读写、Shell 命令执行、解析等可能经常发生且可以预期的失败，系统使用 `Result<TValue, TError>`。
*   **原则**: 绝对不 `throw`。
*   **优势**: 强制调用方处理错误。例如 `ExecutionEnv.readFile()` 如果文件不存在，会返回一个安全的错误包装，而不会把整个 Agent 炸掉。

### 高层编排层：抛出即中止
对于 `Session` 读写和 `AgentHarness` 的高级突变 API（如 `prompt`）。
*   **原则**: 拒绝/抛出异常。
*   **优势**: 当数据库写入失败时，试图返回一个 `Result` 让上层继续假装无事发生是极其危险的。这时候抛出 `SessionError` 让应用停机是最安全的做法。

### 钩子失败策略：勇往直前
如果一个 Hook（例如用于记录日志的 Hook）在 Harness 已经提交了状态变更后突然抛出了错误（比如网络日志服务器挂了），系统该怎么办？
*   **原则**: 状态变更**不回滚**。Harness 会将该错误捕获，标准化为带有 `"hook"` 代码的 `AgentHarnessError`，然后抛给外层。但此时 Agent 的状态已经成功推进到了下一步。

## 💾 保存点 (Save Points) 的微观解剖

保存点是 Harness 混乱与秩序的分界线。它发生在**一个助手回合及其相关的工具结果消息都已成功入库之后**。

在触发保存点的那几毫秒内，Harness 会以极其严格的顺序执行以下步骤：
1. **清空挂起池**: 将 `pendingSessionWrites` 中的所有操作（修改模型、更改提示词等）按顺序冲刷到 `Session` 数据库。
2. **状态评估**: 询问底层的 Loop 配置（`prepareNextTurn` 回调），判断是应该彻底停机，还是需要立刻根据最新的 Live Config 创建一份新的快照，继续下一轮交互。
3. **安全衔接**: 如果需要继续下一轮，新生成的快照会吸收刚才第 1 步中应用的所有配置变更。因此，系统实现了“在飞行中更换引擎”的无缝衔接。

> [!important] 为何要在消息之后刷新？
> 假设 LLM 刚说完一句 "Hello"，然后用户在界面点击了 "切换模型到 GPT-4"。
> 如果先刷新用户的配置，再存消息，数据库的顺序就是：`[切换模型] -> [Hello]`。当你回顾历史时，你会误以为这句 "Hello" 是用 GPT-4 生成的。
> 保存点的顺序机制确保了数据库记录总是：`[Hello (来自旧模型)] -> [切换模型]`，完美保留了历史的因果律。
