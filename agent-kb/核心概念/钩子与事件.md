# 🪝 钩子与事件 (Hooks & Events)

为了满足无穷无尽的业务需求（例如：添加一个安全命令拦截插件、添加一个在请求前自动补全当前目录树作为上下文的插件，或者实现与自定义鉴权系统的对接），单纯的面向对象继承是脆弱且僵化的。`AgentHarness` 架构设计了一套高级的、类型安全的、高度可编排的 Hook（钩子）与 Event（事件）系统。

> [!info] 区别于普通的 EventEmitter
> 传统的 EventEmitter（如 Node.js 中的 `EventEmitter`，或是 DOM 中的 `addEventListener`）大多是只读的“通知-广播”机制。
> 本系统中的 Hooks 是一个支持**异步等待 (Async Await)** 并且核心支持 **数据流突变 (Mutation)** 的处理管道 (Pipeline)。插件不仅能“看”，还能合法地“篡改”数据。

## 🛡️ 类型安全的“幻影结果” (Phantom Result Types)

这是本 Hook 系统的一大工程亮点。为了防止插件作者返回错误的数据结构导致核心系统在深处崩溃，系统巧妙地使用了 TypeScript 的 `symbol` 定义了幻影类型。

每一个事件接口在定义时，不仅定义了它携带的数据，还在类型层面强制绑定了**处理该事件的函数必须返回的结果格式**。

```typescript
// 1. 定义一个全局唯一的符号，它不会在运行时产生任何额外的属性开销
declare const HookResult: unique symbol;

// 2. 基础事件接口，泛型 TResult 用于承载期望的返回值类型
interface HookEvent<TType extends string, TResult = void> {
	type: TType;
	readonly [HookResult]?: TResult; // 这是一个幻影属性，在运行时不存在
}

// 3. 具体的事件定义。例如 tool_call 事件，它期望插件返回是否拦截该工具的决策
interface ToolCallEvent extends HookEvent<"tool_call", { block?: boolean; reason?: string }> {
	type: "tool_call";
	toolName: string;
	input: Record<string, unknown>; // 将传递给工具的参数
}
```

当开发者在外部调用 `harness.on("tool_call", handler)` 时，TypeScript 编译器会自动推导出 `handler` 的返回类型必须是 `{ block?: boolean; reason?: string }` 或者是 `void`。这就从根本上杜绝了类型不匹配导致的运行时异常。

## ⚙️ 处理器的三种执行语义 (Execution Semantics)

当你注册了多个插件（Handler），它们是如何协同工作的？如果两个插件都想修改同一个数据，系统听谁的？

针对不同类型的事件，`AgentHarness` 的内部调度器提供了三种极其明确的数据约简（Reduction）策略：

```mermaid
graph TD
    subgraph 1. 转换链 Transform Chain (适用于上下文注入)
        In1[输入消息流] --> P1[插件A: 添加当前时间]
        P1 -->|修改后流转| P2[插件B: 扫描时间并补充天气]
        P2 --> Out1[最终发给大模型]
    end

    subgraph 2. 补丁累加 Patch Accumulation (适用于工具结果修饰)
        In2[工具原始执行结果] --> M1[插件A: 发现错误, 标记 isError=true]
        In2 --> M2[插件B: 删除敏感输出文本]
        M1 & M2 --> Merge[浅层按字段合并]
        Merge --> Out2[合成的安全新结果]
    end

    subgraph 3. 早期退出 Early Exit (适用于权限控制)
        In3[工具执行请求] --> C1[插件A: 黑名单检查通过]
        C1 --> C2[插件B: 检测到高危 `rm -rf`, 立即返回 block:true]
        C2 -.x|管道熔断| C3[插件C: 性能日志分析 (被跳过)]
        C2 --> Out3[工具被拦截, 直接失败]
    end
```

### 1. 转换链 (Transform Chain)
*   **适用事件**: `context` (处理将发往 LLM 的历史消息列表), `before_provider_payload` (处理 HTTP 请求的底层 JSON 体)。
*   **机制**: 就像一条流水线（Pipeline）。第一个插件处理完数据后，其输出结果会被立刻合并到事件体中，然后传递给第二个插件。第二个插件看到的是已经被第一个插件修改过的**最新状态**。
*   **威力**: 允许插件之间产生联动。

### 2. 补丁累加 (Patch Accumulation)
*   **适用事件**: `tool_result` (工具刚刚执行完毕，正准备写入记录并发送给模型时)。
*   **机制**: 各个插件是“背靠背”提出修改意见的。每个插件都可以返回一个**局部补丁 (Partial Patch)**（例如只修改 `details` 字段，不修改 `content` 字段）。系统引擎会在最后，将这些零散的补丁统一做一次**浅层对象合并 (Shallow Merge)**，覆盖原始的工具结果。
*   **威力**: 多个独立的插件可以同时为一个复杂对象提供修饰，而无需关注对方的存在。如果有针对同一个字段的修改，后执行的插件拥有覆盖权。

### 3. 早期退出 / 短路熔断 (Early Exit / Short-circuit)
*   **适用事件**: `tool_call` (工具即将执行的毫秒前), `session_before_compact` (历史压缩操作即将启动时)。
*   **机制**: 按照插件注册的先后顺序严格依次执行。一旦任何一个插件在返回值中给出了明确的阻断指令（例如 `{ block: true }` 或 `{ cancel: true }`），整个 Hook 管道会立即**熔断**！后续排队的所有插件都将失去执行机会，操作立即被拦截。
*   **威力**: 是构建诸如“安全沙箱”、“权限控制层”等具有一票否决权系统的绝对利器。

## 👁️ 观察者模式 (Observer Pattern)

如果你写插件的目的不是为了“改变世界”，而仅仅是为了“记录世界”（例如：统计一下系统一天调用了多少次大模型，每次用了多少毫秒），你应该使用纯粹的观察者模式。

*   **注册方式**: 使用特权的全局监听器或调用订阅 API。
*   **核心特性**: 观察者只能接收到最初发出的**原始事件副本**。无论排在后面的转换链插件如何疯狂地篡改数据，观察者看到的数据都不会被污染。同时，TypeScript 会忽略观察者的任何返回值，这在编译器层面阻止了日志系统意外干扰核心业务逻辑的可能。这确保了可观测性系统（Observability）的绝对客观与公正。

> [!example] 面试与实战推演
> 假设你需要开发一个功能：**当模型生成特定的代码时，立刻自动调用系统的 Linter（代码检查工具），并将 Linter 的报错信息附加在回答后面**。
>
> *架构思路：你可以不碰任何 `AgentHarness` 的源码。只需注册一个对 `turn_end` 或 `message_end` 的观察者事件来捕获模型刚生成的文本。同时利用 Harness 提供的 `followUp()` API，将含有报错信息的 Linter 结果以系统消息的形式推入后续队列。这样，Agent 会在下一个回合自动看到这些错误并尝试修正。这完美体现了事件系统与队列系统的协同之美。*
