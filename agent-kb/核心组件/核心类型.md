# 🧱 核心类型 (Core Types)

在 TypeScript 架构的大型复杂系统中，类型定义往往比代码实现更能说明系统设计的边界与契约。`packages/agent` 采用了强类型系统，将各种松散耦合的组件像拼图一样严密地咬合在一起。

## 📍 定义位置
- 核心引擎类型：`packages/agent/src/types.ts`
- 编排器特有类型：`packages/agent/src/harness/types.ts`

## 💬 消息系统 (Message System)

系统中的消息并不是简单的字符串，而是带有极强上下文语义的结构体。

### `AgentMessage`
这是系统内部流转的唯一消息标准。它是一个**联合类型 (Union Type)**。除了包含底层 API 库（`@earendil-works/pi-ai`）规定的标准消息类型外：
- `UserMessage`
- `AssistantMessage`
- `ToolResultMessage`

它还支持应用层的特殊扩展。

### `CustomAgentMessages` 与声明合并
这是一个非常巧妙的设计模式。核心包并不知道外部的 CLI 或 Web 界面需要发送什么特殊消息（比如进度条通知、或者是一个特殊的错误堆栈）。
通过导出一个空接口 `CustomAgentMessages`，外部应用程序可以使用 TypeScript 的**声明合并 (Declaration Merging)** 特性，安全地把自己的类型塞入系统的联合类型中，而不需要修改核心包的任何代码。

```typescript
// 核心包中定义为空
export interface CustomAgentMessages {}

// ------

// 在你的外部业务代码中，你可以随时这样扩展它：
declare module "@mariozechner/agent" {
  interface CustomAgentMessages {
    // 扩展一个自定义的工单卡片消息
    ticketCard: { role: "custom", type: "ticketCard", ticketId: string };
  }
}

// 这样，AgentMessage 类型就自动包含了 ticketCard 类型，且受到完全的编译期检查保护。
```

## ⚙️ 执行配置的依赖注入容器

### `AgentLoopConfig`
由于底层的 `AgentLoop` 被设计为一个极度纯净的函数库（没有任何平台依赖），`AgentLoopConfig` 就成了它获取外部能力的唯一途径。它本质上是一个**依赖注入 (Dependency Injection) 容器**。

关键契约包括：
- `model`: 模型定义对象。
- `convertToLlm`: 这是一个核心适配器函数。因为上述的 `AgentMessage` 中可能包含了各种各样花里胡哨的 `CustomAgentMessages`（大模型 API 显然不认识它们）。这个函数负责将内部上下文**降级并清洗**为各大模型厂商特定的纯净 `Message` 格式。通常，在这里会将自定义的 UI 消息剔除，或者转化为纯文本备注。
- `transformContext`: 在清洗发生前，允许进行最终的干预。通常用来进行**窗口滑动截断**（比如删除 100 轮之前的废话，防止超出 Token 限制）。
- `prepareNextTurn`: 这是一个极具战略意义的回调。它发生在工具执行完毕后、发起下一次大模型网络请求前。在 `AgentHarness` 中，正是利用这个回调触发了 `flushPendingSessionWrites()`，把挂起的配置安全落盘。

### `AgentTool` 的扩展语义
普通的 `Tool` 只需要定义名字和参数 Schema。但在我们系统中，`AgentTool` 增加了一套完整的执行语义：

```typescript
export interface AgentTool<TParameters extends TSchema = TSchema, TDetails = any> extends Tool<TParameters> {
	// ...
	
	// 容错器：大模型有时候给出的 JSON 不太符合 Schema 规范（比如把布尔值写成了字符串）。
	// 这个函数可以在参数真正进入校验器之前，做一次兼容性的预处理和修复。
	prepareArguments?: (args: unknown) => Static<TParameters>;
	
	// 真正的战壕：执行函数。
	// 它不仅接收参数，还接收一个 `onUpdate` 回调。
	// 工具可以通过频繁调用 onUpdate，将部分结果实时流式推送到前端（比如绘制一个长耗时的下载进度条）。
	execute: (
		toolCallId: string,
		params: Static<TParameters>,
		signal?: AbortSignal,
		onUpdate?: AgentToolUpdateCallback<TDetails>,
	) => Promise<AgentToolResult<TDetails>>;
	
	// 隔离性标记。如果是会搞乱文件系统的危险工具，必须标记为 sequential，拒绝并发。
	executionMode?: ToolExecutionMode;
}
```

## 📡 事件系统 (Event System)

系统中存在两个维度的事件。理解它们的差异是掌握系统扩展机制的关键。

### `AgentEvent` (底层纯净事件)
这是底层 `runLoop` 执行时像打字机一样吐出的扁平化纯记录事件。它们就像是监控录像带。
- **特点**: 没有 Hook 修改机制，只是单纯地告诉你“发生了什么”。
- **分类**: 
  - 骨架阶段：`agent_start`, `agent_end`, `turn_start`, `turn_end`
  - 碎片流动阶段：`message_start`, `message_update`, `message_end`
  - 虚拟动作阶段：`tool_execution_start`, `tool_execution_update`, `tool_execution_end`

### `AgentHarnessEvent` (高层业务突变事件)
这是建立在 `AgentHarness` 层面的业务级事件。它们构成了 Hook 系统的基石。
- **特点**: 这类事件通常伴随着期待的返回类型（幻影返回类型），允许插件拦截并修改系统走向。
- **分类**:
  - **拦截点 (Interceptors)**: `context`, `tool_call` (决定是否执行), `tool_result` (篡改执行结果), `before_provider_request` (修改发往服务器的底层 HTTP Headers，如加上代理参数)。
  - **里程碑 (Milestones)**: `save_point` (告诉你所有配置都落盘了，可以刷新页面了), `model_select`, `abort` 等。
