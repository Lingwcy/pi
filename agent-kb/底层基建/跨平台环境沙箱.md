# 📦 跨平台环境沙箱 (Execution Environment)

在开发 Node.js 应用时，如果要读取一个文件，绝大多数开发者的本能反应是直接在文件头部写上：
`import * as fs from 'fs';` 或 `import { exec } from 'child_process';`

然而，在 `packages/agent` 的核心代码（包括 Harness 和 Loop）中，你**绝对找不到**任何一行直接引入这些系统模块的代码。

为什么？因为一旦引入了 `fs`，这个 Agent 就永远被锁死在 Node.js 环境中了。它将无法在浏览器端的 WebContainer 中运行，也无法在 Cloudflare Workers 这种边缘计算环境中部署。

## 🛡️ 依赖倒置原则 (Dependency Inversion Principle)

为了彻底解耦，系统在 `src/harness/types.ts` 中定义了一个极度纯净的接口：`ExecutionEnv`。

所有 Agent 需要与现实世界打交道的能力，全部被抽象为这几个异步方法：

```typescript
export interface ExecutionEnv {
    // 文件操作
    readFile(path: string): Promise<Result<Uint8Array, EnvError>>;
    writeFile(path: string, content: Uint8Array): Promise<Result<void, EnvError>>;
    appendFile(path: string, content: Uint8Array): Promise<Result<void, EnvError>>;
    
    // 执行能力
    execShell(command: string, options?: ExecOptions): Promise<Result<ExecOutput, EnvError>>;
    
    // 其他基础能力
    setTimeout(ms: number): Promise<void>;
}
```

Agent 核心代码在需要读文件时，只会调用 `this.env.readFile()`。至于这个文件是从真正的硬盘读的，还是从浏览器的 IndexedDB 读的，Agent 根本不关心。

## 🏗️ 宿主注入与运行时适配

真正的活儿，交给了宿主程序（Host Application）去干。

在 `src/harness/env/nodejs.ts` 中，系统提供了一个基于 Node.js `fs` 和 `child_process` 的标准实现类 `NodeExecutionEnv`。

当你在 CLI 或后端启动应用时，你需要手动把它注入给 Harness：
```typescript
import { NodeExecutionEnv } from "@earendil-works/pi-agent/env/nodejs";

const harness = new AgentHarness({
    env: new NodeExecutionEnv(), // 依赖注入发生在这里
    session: ...,
});
```

如果你想开发一个纯前端的 VSCode 插件（Web Extension），你只需要自己实现一个 `VSCodeWebEnv implements ExecutionEnv`，底层调用 VSCode 的 `vscode.workspace.fs` API，Agent 就能在浏览器里跑起来了！

## 🔒 意想不到的红利：安全审计与虚拟化

这种沙箱设计不仅解决了跨平台问题，还带来了巨大的安全和测试收益：

### 1. 拦截与鉴权
由于所有的敏感操作都必须经过 `ExecutionEnv`，宿主程序可以在实现这个接口时做手脚。
例如，实现一个 `RestrictedEnv` 包装器：
```typescript
class RestrictedEnv implements ExecutionEnv {
    constructor(private baseEnv: ExecutionEnv, private allowedDir: string) {}

    async readFile(path: string) {
        if (!path.startsWith(this.allowedDir)) {
            return err("安全拦截：禁止读取沙箱外文件");
        }
        return this.baseEnv.readFile(path);
    }
}
```
通过极其优雅的装饰器模式，Agent 就被关进了一个无法越狱的文件监狱里。

### 2. 测试虚拟化 (In-Memory Mocking)
在编写单元测试时，如果每次都要去真的写文件、删文件，测试不仅跑得极慢，而且容易留下垃圾数据。
有了 `ExecutionEnv`，测试框架只需实现一个 `MemoryEnv`，用一个 `Map<string, Uint8Array>` 来模拟整个文件系统。所有的测试都能在毫秒级内完成，且彼此绝对隔离。
