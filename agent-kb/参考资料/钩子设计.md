# 🪝 钩子系统设计 (详细参考)

在研究了 [[核心概念/钩子与事件]] 的理论之后，这篇参考资料将带你深入探究这个系统在代码层面是如何设计与实现的。理解这份文档，你将能够自行开发出极其复杂的 Agent 扩展插件。

## 💡 为什么抛弃 EventEmitter？

Node.js 社区对 `EventEmitter` 非常熟悉：`emitter.on('event', data => console.log(data))`。
但在 AgentHarness 中，如果使用 EventEmitter，会遇到几个致命缺陷：
1. **无法控制顺序**：多个监听器的执行顺序是未定义的。
2. **无法异步等待**：如果一个拦截器需要花 2 秒钟去查询远程数据库进行权限校验，EventEmitter 无法等待它。
3. **无法突变数据流**：监听器只能收到数据的副本，无法将修改后的数据优雅地传递给下一个监听器，更无法将结果返回给事件的触发者。

因此，Harness 的 Hooks 被设计成了一个**异步的、强类型的、基于责任链模式的处理器集合**。

## 🧬 核心模型：幻影类型的魔法 (Phantom Types)

在 TypeScript 中，事件本身是一个纯数据对象。我们如何让事件对象“携带”对处理它的函数的返回值类型的约束呢？

通过定义一个唯一且不可枚举的 Symbol，我们可以创建一个在编译期存在、而在运行时隐形的“幻影属性”。

```typescript
// 1. 这个 Symbol 是私有的，防止外部意外覆盖
declare const HookResult: unique symbol;

// 2. 泛型基类，TResult 默认为 void，代表大多数事件不需要返回值
interface HookEvent<TType extends string, TResult = void> {
	type: TType;
	readonly [HookResult]?: TResult;
}

// 3. 提取事件期望返回值的类型体操
// 如果 E 这个事件对象里面有 [HookResult] 属性，就推导出它的类型 R，否则就是 void
type ResultOf<E> = E extends { readonly [HookResult]?: infer R } ? R : void;

// 4. 定义处理器的类型。它的返回值必须严格等于该事件推导出的 ResultOf<E>
type HookHandler<E, Ctx> = (
	event: E,
	ctx: Ctx,
	signal?: AbortSignal,
) => ResultOf<E> | void | Promise<ResultOf<E> | void>;
```

这就解释了为什么当你在 VSCode 中输入 `harness.on("tool_result", (event) => { return ... })` 时，编辑器能精准地告诉你，你只能返回包含 `content`, `details`, `isError` 属性的对象。

## 🏛️ `AgentHarnessHooks` 接口规范

整个钩子系统的核心接口定义得极其克制：

```typescript
interface AgentHarnessHooks<E extends HookEvent<string, unknown>, Ctx> {
	// 供只读记录使用的旁观者
	observe(handler: HookObserver<E, Ctx>): () => void;

	// 真正的参与者。类型系统保证了 type 和 handler 事件类型的强对应
	on<TType extends E["type"]>(
		type: TType,
		handler: HookHandler<Extract<E, { type: TType }>, Ctx>,
	): () => void;

	// 这是唯一供 Harness 核心调用的方法！触发事件并汇总返回值
	emit<TEvent extends E>(
		event: TEvent,
		signal?: AbortSignal,
	): Promise<ResultOf<TEvent> | undefined>;
}
```

注意：`emit` 是唯一暴露给系统内部用来派发事件的方法。Harness 内部**绝对不会**去自己维护监听器数组或者自己写循环。所有的复杂性都被封装进了 Hooks 的实现类中。

## 🛠️ 变异语义的具体实现源码 (The Reducers)

不同的事件，`emit` 函数内部收集返回值的逻辑（Reducer）截然不同。以下是系统核心变异模式的伪代码实现：

### 模式 A: 转换链 (用于 context)
**目标**：串行修改，将上一次的输出作为下一次的输入。

```typescript
let currentEvent = originalEvent;

// 按照注册顺序，依次执行该事件类型下的所有 handler
for (const handler of handlers("context")) {
    const result = await handler(currentEvent, ctx, signal);
    
    // 如果插件修改了 messages，创建一个新对象，作为下一个插件的输入
    if (result?.messages) {
        currentEvent = { ...currentEvent, messages: result.messages };
    }
}

// 最后，将最终被修改过的 messages 返回给 Harness
return { messages: currentEvent.messages };
```

### 模式 B: 补丁累加 (用于 tool_result)
**目标**：每个人都可以修饰对象的一部分，最后统一合并。

```typescript
let currentPatch = originalEvent;

for (const handler of handlers("tool_result")) {
    const patchResult = await handler(currentPatch, ctx, signal);
    if (!patchResult) continue;

    // 浅层合并。如果有冲突，后执行的 handler 拥有决定权
    currentPatch = {
        ...currentPatch,
        content: patchResult.content ?? currentPatch.content,
        details: patchResult.details ?? currentPatch.details,
        isError: patchResult.isError ?? currentPatch.isError,
    };
}
return currentPatch; // 返回合并后的完整补丁
```

### 模式 C: 早期退出 (用于 tool_call 拦截)
**目标**：一票否决。

```typescript
for (const handler of handlers("tool_call")) {
    const result = await handler(event, ctx, signal);
    
    // 一旦发现有插件返回了 block: true
    if (result?.block) {
        // 直接 break 掉循环，立刻把这个决定返回给 Harness
        // 后面排队的插件连执行的机会都没有
        return result; 
    }
}
return undefined; // 大家都没意见，放行
```

## ⚠️ 扩展开发者的错误政策约束

如果你在开发扩展时代码抛出了异常，默认的策略是**继续执行 (Continue)**，这在原始设计文档中被强调：
`errorMode: "continue" | "throw"`

Harness 不希望因为一个提供天气格式化的第三方小插件因为网络超时抛出异常，就导致耗费了大量 Token 的大模型思考过程整个崩溃。
因此，`emit` 内部通常会使用 `try-catch` 包裹第三方处理器的执行。如果插件崩溃，系统会记录日志，然后假装它没有返回任何变异数据，继续把事件传给下一个健康的插件处理。

这要求核心库在异常容忍度上做到极致的防弹设计。
