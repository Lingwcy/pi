# 🧩 Harness 子系统 (Subsystems)

为了维持 `AgentHarness` 主类的代码边界与轻量化，许多复杂的高级逻辑（如管理长上下文、历史溯源、动态提示词生成）被剥离出来，形成了独立的子系统。它们通常位于 `src/harness/` 目录下的独立模块中，由 Harness 在相应的生命周期节点按需调用。

## 1. 历史压缩引擎 (Compaction Engine)
**位置**: `src/harness/compaction/compaction.ts`

当大语言模型面对几十轮、长达上万 Token 的上下文时，它不仅会变得迟钝，API 的计费也会随之爆炸。更糟糕的是，如果超出了模型的最大 Context Window，对话将直接报错崩溃。

> [!info] 为什么不能直接截断数组？
> 你不能简单地通过 `messages.slice(-5)` 来保留最后 5 轮对话。因为早期的对话中可能包含极其关键的信息（例如：“系统的密码是 1234”、“不要使用 rm 命令”）。如果盲目丢弃，模型会产生失忆。

历史压缩引擎提供了一种极其优雅的“无损降级”机制。

### 🔪 智能切割点算法 (`findCutPoint`)
这是引擎最核心的算法。它的任务是：从长长的历史中，找到一刀切下去最合适的那个“节点”。
1. **反向计算预算**: 算法接收一个 `keepRecentTokens`（例如，无论如何要保留最近的 20,000 个 Token 原汁原味）。它会从最新的对话往前推算，直到累计 Token 达到阈值。
2. **寻找“安全地带”**: 算法绝不会在刚刚好达到预算的地方切断。它会继续往前找，直到遇到一个完美的**回合边界 (Turn Boundary)**。
   *   它不会把一个漫长的 Assistant 推理流从中间切断。
   *   它绝不允许把一次 `ToolCall` 和其对应的 `ToolResult` 强行拆散。
3. **输出切割点**: 返回 `CutPointResult`，指明哪条记录将被作为新时代的开端。

### 📝 LLM 摘要化重塑 (`generateSummary`)
确定了切割点后，引擎会将那些“即将被丢弃”的古老对话，全盘发给大模型本身。
它注入了一个硬编码的极强指令 `SUMMARIZATION_SYSTEM_PROMPT`，强迫模型按照极其严苛的 Markdown 格式输出摘要：
*   **Goal**: 用户当初的最终目的是什么？
*   **Progress (Done/In Progress/Blocked)**: 任务推进到了哪一步？
*   **Key Decisions**: 那些曾经犯过的错和做出的关键架构决策。

### 💾 虚拟节点固化
生成的摘要不会凭空消失。Harness 会在 Session 数据库的切割点之前，插入一个特殊的虚拟节点：`CompactionEntry`。
在未来向模型发送请求时，底层转换器（`convertToLlm`）会识别到这个条目，并将其转换成一条隐形的 `System Message` 悄悄放在开头。这样，模型在“失忆”后，总能先读到这本属于过去的“备忘录”。

## 2. 分支时光机 (Branch Summarization)
**位置**: `src/harness/compaction/branch-summarization.ts`

这个功能让 Agent 的对话历史拥有了类似 Git 版本控制的能力。

> [!example] 应用场景
> 假设你正在让 Agent 解决一个 Bug：
> - **节点 A**: 开始解决 Bug。
> - **分支 1 (节点 B->C)**: Agent 决定使用正则匹配去替换，修改了 3 个文件。结果发现正则表达式过于复杂，测试全面失败。
> - **时间回溯**: 你很不满，在界面上点击**节点 A**，选择“从此节点继续（分支切换）”。
> 
> 在过去，分支 1 的失败尝试就永远随风而逝了。但在新的分支上，Agent 极大概率会重蹈覆辙，再次尝试使用正则。

### 🧬 分支摘要算法的魔法
当调用 `navigateTree(targetId, { summarize: true })` 时：
1. **LCA (最近公共祖先) 寻路**: 算法会对比当前所处的叶子节点（分支 1 的尽头）和你点击的目标节点（节点 A），在图论上找到它们的最近公共祖先（LCA）。
2. **废弃物回收**: 算法将从 LCA 一路向下直到分支 1 尽头的所有对话、以及在这个过程中 Agent **涉及修改过的所有文件路径 (FileOperations)** 收集起来。
3. **经验总结**: 发送给 LLM，要求其写出一份“失败复盘报告”。
4. **经验注入**: 系统会在你新开的**节点 A** 之后，插入一个 `branch_summary` 节点。
   
这样，当 Agent 在新分支上重新思考时，它第一眼看到的就是：“*系统警告：你曾在此处的另一条时间线上探索过。你尝试了修改这 3 个文件并使用了正则，结果失败了。请吸取教训，更换策略。*” 
这是赋予 AI **长期反思能力**的神来之笔。

## 3. Skills (技能体系)
**位置**: `src/harness/skills.ts`

不要把 Skills（技能）和 Tools（工具）混为一谈。
- **Tool (工具)** 是提供给模型调用的函数能力（例如“读取文件”、“执行命令行”）。
- **Skill (技能)** 是**经验的打包**。它本质上是一段封装好的、高质量的系统提示词（System Prompt），外加它所依赖的一组特定工具。

当你在 UI 上调用 `harness.skill("CodeReviewer")` 时：
系统并不会真的执行一个叫 CodeReviewer 的函数。Harness 会查阅库中的技能配置，将它转化为一段强有力的指导语，将其置入 Agent 的脑海，并自动点亮它需要的审查工具池，引导 Agent 按照预设的专家角色开始工作。

## 4. Prompt Templates (提示词模板库)
**位置**: `src/harness/prompt-templates.ts`

这是用来减少重复劳动的快捷指令系统。
如果你每天都需要检查代码风格，你不需要每次都手打长篇大论。
系统支持带有 `{{变量名}}` 占位符的模板机制。调用 `promptFromTemplate("check_style", ["src/index.ts"])` 时，子系统会安全地将参数渲染进模板中，然后透明地将其当作用户的输入传递给大模型启动回合。
