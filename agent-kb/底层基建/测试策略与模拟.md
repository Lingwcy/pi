# 🧪 测试策略与模拟 (Testing Strategy & Faux Provider)

测试一个普通的纯函数（比如 `add(a, b)`）非常简单。但当你面对一个融合了**网络流、并发锁、重试机制、以及完全不可控的大模型 API**的 Agent 系统时，传统的测试方法将彻底失效。

如果你的单元测试直接调用 OpenAI 的 API，你将面临三个噩梦：
1. **慢**：跑完一套覆盖所有分支的测试可能需要半小时。
2. **贵**：每次跑 CI/CD 都在疯狂燃烧美金。
3. **脆弱 (Flaky)**：大模型今天回答 A，明天回答 B；有时候还会报 502 网络错误。你的测试会随机失败，导致团队彻底失去对测试的信任。

`packages/agent` 是如何驯服这个混沌巨兽的？答案是**极致的确定性模拟 (Deterministic Mocking)**。

## 🎭 核心武器：Faux Provider (伪造提供商)

系统设计了一套名为 `Faux Provider`（在 `packages/ai` 中定义，在 agent 测试中被广泛使用）的模拟框架。
它不仅仅是拦截 HTTP 请求那么简单，它**深度模拟了大模型流式生成的微观物理规律**。

### 1. 模拟流的“碎片化”
为了测试底层的 `streamAssistantResponse` 状态机（见 [[核心组件/AgentLoop函数]]）是否能正确拼接 JSON，Faux Provider 可以精确控制怎么吐出碎片：

```typescript
// 伪代码：在测试脚本中定义大模型该怎么表现
const fauxModel = registerFauxProvider({
    // 当收到任何 prompt 时，强制它模拟调用一个名为 "read_file" 的工具
    responseFactory: (request) => {
        return fauxStream([
            { type: "text_delta", content: "I will read " },
            { type: "text_delta", content: "the file now." },
            { type: "toolcall_start", id: "call_1", name: "read_file" },
            { type: "toolcall_delta", content: `{"pa` }, // 故意把 JSON 撕碎
            { type: "toolcall_delta", content: `th": "a.txt"}` },
            { type: "done" }
        ]);
    }
});
```
这种极致的控制粒度，使得测试用例可以故意制造一些大模型常常犯的错（比如 JSON 截断），来验证核心引擎的鲁棒性。

### 2. 模拟网络灾难与 Abort
如何测试生命周期中那个极其复杂的 `harness.abort()` 是否真的能切断网络、且不丢失数据？
测试脚本可以这样写：
1. 配置 Faux Provider，在吐出第三个字符时，故意暂停 `await sleep(5000)`。
2. 调用 `harness.prompt()`。
3. 在测试主线程里立刻调用 `harness.abort()`。
4. 断言：底层的 Promise 是否立刻抛出了 Abort 错误？持久化队列是否仍然被正确冲刷（Flush）了？

## 🛡️ 可重入性与恶意 Hook 测试 (Hook Reentrancy)

官方文档中特别强调了对于 Harness 的 "Hook reentrancy test suite"（钩子可重入性测试套件）。

> [!danger] 什么叫可重入性破坏？
> 假设你写了一个插件挂载在 `tool_call` 事件上。由于这个 Hook 是 Harness 在 `turn` 忙碌阶段触发的，如果这个插件在自己的代码里，又去调用了 `harness.prompt("新的问题")`，这就是**重入**。
> 如果系统没有良好的锁机制，这会立刻导致状态机崩溃。

因此，测试策略中包含了一组“恶意行为测试”：
- 测试脚本注册一个 Hook，该 Hook 试图在接收事件时修改系统的模型配置。
- 测试脚本注册一个 Hook，试图在 `context` 转换时抛出语法异常。
- 测试断言：系统是否正确地返回了 `busy` 错误？系统是否捕获了插件的异常并将其标准化为带有 `hook` code 的 `AgentHarnessError`，且**保证了当前的回合没有因为插件的崩溃而回滚**？

## 💾 持久化边界的快照测试

为了验证复杂的“挂起写入队列”是否真的按顺序落盘了，系统不使用真实的数据库，而是注入一个 `MemoryRepo`（基于内存的持久化仓库）。
测试流程：
1. 启动 Agent。
2. 疯狂注入并发修改指令。
3. 等待 `settled` 事件触发。
4. **快照断言 (Snapshot Assertion)**：直接导出 `MemoryRepo` 中的 JSONL 数组，与硬编码在测试里的期望数组结构进行精确比对，验证 `model_change` 记录是否严丝合缝地插在了两条对话消息之间。

> [!abstract] 软件工程感悟
> “一段不能被轻易测试的代码，就是一段已经腐烂的代码。”
> `packages/agent` 之所以要把 `ExecutionEnv` 抽离出来，之所以要不厌其烦地设计 Faux Provider，所有的努力都是为了一个终极目标：**让业务逻辑与外部依赖解耦，从而获得 100% 的单元测试确定性。** 这种基于接口而非实现的编程范式（IoC），是构建大型可靠系统的生命线。
