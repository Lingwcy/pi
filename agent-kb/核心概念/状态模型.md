# 🧊 状态模型 (State Model)

`AgentHarness` 系统能够做到在极度复杂的并发环境下依然坚如磐石，归功于其独创的多层次状态模型。这主要为了解决 AI 交互场景中一个致命的工程问题：**并发配置修改与执行时间线的对齐**。

> [!question] 为什么需要并发控制？
> 传统的 Web 后端请求大多是毫秒级的。但在 AI Agent 的场景中，Agent 长达数十秒甚至几分钟的思考和工具调用（如正在执行一个复杂的 Shell 脚本）是一个极为漫长的“阻塞期”。
> 在这个漫长的时间窗口内，用户可能随时在 UI 上点击按钮修改配置（比如：切换当前正在使用的大模型、关闭某个之前开启的工具、或者打上了一个标记）。
> 如果系统收到用户的修改指令后立刻**直接应用**，会导致底层的执行请求出现严重竞态而崩溃；如果立刻**直接写入数据库日志**，则会导致在历史记录的时间线上，这次“模型切换记录”荒谬地插在了一条还未生成完毕的回复的正中间。

## 🧅 状态的四个层级：洋葱式隔离机制

为了彻底解决竞态和时间线错乱问题，系统设计了如同洋葱般的四层状态隔离与渗透机制：

```mermaid
graph TD
    User((用户/UI操作)) -->|实时修改 setModel()| LiveConfig[1. 活配置 Live Config]
    LiveConfig -.->|在 prompt() 回合开始时\n创建克隆| Snapshot[2. 回合快照 Turn Snapshot]
    
    Snapshot -->|作为唯一的真相来源传入| AgentLoop(AgentLoop 执行引擎)
    
    User -.->|如果当前正处于 turn 忙碌阶段\n拦截写入| PendingQ[4. 待处理写入队列 Pending Writes]
    AgentLoop -->|产生消息等确定性事实| Session[(3. Session 树状持久化层)]
    
    PendingQ -.->|在回合结束的 安全保存点| Session
```

### 1. Harness 内存配置 (Live Config)：保证 UI 响应的极致流畅
这是内存中最顶层、最实时的状态。
- **包含内容**: 当前选定的模型 (Model)、系统的思考级别 (ThinkingLevel)、系统当前装载的工具池及激活列表。
- **行为准则**: 极度响应性。无论系统当前是否正在狂飙突进地生成文本，外部调用诸如 `harness.setModel("gpt-4o")` 都会**立即无条件更新**这里的内存值。这意味着，前端 UI 如果从 Harness 读取当前状态，得到的永远是用户刚刚点击的那个结果，不会有任何卡顿或延迟感。

### 2. 回合快照 (Turn Snapshot)：保障底层执行的绝对确定性
这是整个防并发系统的定海神针。
- **机制**: 在每次调用 `harness.prompt()` 开始一个新的对话回合的毫秒间，Harness 会对那一瞬间的“活配置”进行一次**浅拷贝 (Shallow Copy)**，生成一个封闭的对象 `TurnSnapshot`。
- **作用域**: 底层的狂暴引擎 `AgentLoop` **只认识并且只能访问这个快照**。
- **神迹时刻**: 假设在时间 `T=0` 时，快照捕获到的模型是 `Claude 3.5`。在 `T=5` 秒时，用户疯狂点击 UI 将内存里的 Live Config 改成了 `GPT-4`。由于 `AgentLoop` 手里拿着的是 `T=0` 时的快照，它在接下来的几十分钟内（甚至包括它自行触发的数次重试和工具循环调用），依然会极其稳定地继续使用 `Claude 3.5` 完成当前任务。**当前回合绝不被外界并发污染**。

### 3. 会话记录 (Session / 持久化层)：系统的不变真理
代表系统最终承认的**历史事实**，也是一切恢复和回滚的基础（通常对应着磁盘上的 JSONL 文件或 SQLite 数据库）。
- **包含内容**: 树状链接的对话历史记录 (Messages)、发生过的模型变更标记、当前时间轴的叶子节点指针 (Leaf ID)。
- **操作准则**: **极其苛刻的写入时机**。只有当明确某个事件已经成为不可挽回的历史事实（例如，底层模型通过网络确定地返回了 `message_end` 标志，告知这句话确实说完了），这个事实才会被追加写入 Session。

### 4. 待处理写入队列 (Pending Session Writes)：解决时间线冲突的缓冲带
这是连接上面第一层和第三层的重要缓冲枢纽。
- **场景还原**: 还是回到那个场景，Agent 正在说话的中途，用户在界面上把模型切换成了 `GPT-4`。Live Config 更新了，但正如前文所述，我们绝对不能在这个时间点把这个配置变更写进 Session 数据库里，否则时间线就乱了。
- **挂起机制**: 
  - 当 Harness 检测到自己处于非空闲的忙碌阶段（即 `phase !== "idle"`），所有企图改变持久化层配置的操作（如 `setModel`）都会被拦截。
  - Harness 不会拒绝该操作，而是会生成一张“欠条”（一个记录动作的对象，例如 `{ type: "model_change", provider: "openai", modelId: "gpt-4o" }`），将其塞进名为 `pendingSessionWrites` 的数组中缓存起来。
  - **💾 安全保存点 (Save Points) 的大合唱**: 当漫长的一轮回合最终结束，大模型闭嘴，系统准备回到闲置状态前。Harness 会进入安全的清理环节，调用 `flushPendingSessionWrites()`。此时，时间线是平稳的。Harness 会打开抽屉，拿出所有的“欠条”，依照用户操作的先后顺序，一次性全部刷新到持久化层中。

---

> [!success] 架构带来的巨大工程红利
> 这种“**活配置实时响应 UI请求 + 静态快照隔离执行环境 + 队列延后对齐落盘**”的三维组合架构，彻底消灭了繁琐的并发锁（Mutex）。它使得应用层的 UI 开发者可以毫无心理负担地在任何时候更新业务状态；同时也让底层的核心算法工程师可以永远假设自己运行在一个绝对确定、无竞态条件的理想数学环境中。
