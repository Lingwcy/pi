# 🌳 对话树与持久化引擎 (Session Storage & Repository)

大多数人写 AI 聊天机器人时，会把历史记录存成一个简单的 JSON 数组 `[msg1, msg2, msg3]`。
但在专业的 Agent 系统中，这种做法是灾难性的。为什么？

1. **分支需求**：如果 Agent 写了一段代码，运行失败。用户说“退回上一步，换个方法重写”。此时历史记录发生了**分叉**，简单的数组无法表示平行的多条时间线。
2. **写锁问题**：大模型生成一段长回答可能需要 60 秒。如果每次生成完毕都要把整个几十 KB 的 JSON 数组重写一遍到磁盘，不仅性能极差，而且极易发生覆盖冲突。

`packages/agent` 的 `Session` 系统通过**树状结构** + **JSONL 追加日志** 完美解决了这些问题。

## 🪵 数据结构：不可变的消息树 (Immutable Tree)

系统在内存中将历史记录维护成一棵树。每一个记录节点（无论是用户说的话、大模型生成的文本、还是模型切换的配置变更）都继承自 `SessionTreeEntry` 接口，并且拥有一个关键属性：`parentId`。

```typescript
interface SessionTreeEntry {
    id: string;          // 节点的唯一 UUID
    parentId: string | null; // 指向父节点的 UUID。如果是根节点则为 null
    type: "message" | "model_change" | "compaction" | ...;
    timestamp: number;
}
```

### 树结构的优势
- **极低成本的分支切换**：当用户点击界面上的某句历史对话，说“从这里重新生成”时，系统不需要删除任何旧数据。它只需将当前的“叶子节点指针 (Leaf ID)”指向那句话的 UUID。之后新生成的消息，其 `parentId` 都会指向那个旧节点，自然长出了一条新的分叉。
- **永不丢失的历史**：由于数据永远不被删除，Agent 的每一次失败尝试都被保留在其他分支中，这就为后来的 [[核心组件/Harness子系统#分支时光机 (Branch Summarization)|分支摘要生成]] 提供了素材。

## 📜 存储介质：为什么是 JSONL (JSON Lines)？

`AgentHarness` 的磁盘持久化交给了 `jsonl-storage.ts`。
JSONL 格式意味着文件里的每一行都是一个独立的、合法的 JSON 对象。

```json
{"id":"uuid-1","parentId":null,"type":"message","message":{"role":"user","content":"Hi"}}
{"id":"uuid-2","parentId":"uuid-1","type":"model_change","modelId":"gpt-4"}
{"id":"uuid-3","parentId":"uuid-2","type":"message","message":{"role":"assistant","content":"Hello"}}
```

### Append-only Log (仅追加日志) 的威力
1. **极致性能**：当大模型说出新的一句话时，系统只需要用 Node.js 的 `fs.appendFile` 在文件末尾加一行即可。不需要把整个文件读入内存，也不需要解析整个 JSON 树。时间复杂度为 O(1)。
2. **崩溃安全 (Crash Safe)**：这是核心原因！如果系统在向磁盘写入第 10 句话写到一半时突然断电，JSON 结构被破坏。重启后，系统逐行读取 JSONL，当读到第 10 行时发现 JSON 格式错误，直接丢弃该行即可。前 9 行的数据毫发无损。如果使用一个大 JSON 数组，少了一个括号整个文件都会报废。

## 🔄 重建宇宙：系统重启时的重放算法 (Replay)

当应用重启，Harness 启动时，内存里是空的。它是如何恢复对话现场的？

这套机制借鉴了事件溯源 (Event Sourcing) 或 Redux 的 Reducer 概念：
1. **按行读取**：系统逐行读取 `session.jsonl` 文件。
2. **内存建图**：将每一行解析为一个对象，并利用 `id` 和 `parentId`，在内存中把这棵树原样拼装起来（使用 Map 建立父子索引）。
3. **定位当前分支**：文件中会有特殊的 `leaf` 类型的 Entry，记录着用户最后关心的那个叶子节点的 ID。系统沿着这个叶子节点，顺着 `parentId` 一路回溯到根节点，提取出的这条链（Chain），就是当前对话的**绝对时间线**。
4. **状态归约 (Reduction)**：遍历这条链。如果链上有一个 `model_change`（切换到 Claude），后面又有一个 `model_change`（切换到 GPT-4），系统通过累加计算，最终得出“当前应该使用 GPT-4”的结论，并据此恢复 Harness 的活配置 (Live Config)。

> [!abstract] 架构启发
> 这种以不可变的追加日志作为真相唯一来源 (Single Source of Truth) 的设计，是现代分布式系统和高性能数据库（如 Redis 的 AOF，Git 的 Commit Tree）的基石。应用在 Agent 系统中，赋予了系统极强的抗风险能力和追溯能力。
